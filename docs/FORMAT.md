# `.vip` file format

The canonical save format for viper songs. Plain text, line-oriented,
whitespace-tolerant, diffable, greppable, `:%s`-able.

## Design principles

1. **Round-trip lossless.** In-memory `Song` ↔ `.vip` text must be exactly
   reversible. If you parse then serialize, you get byte-identical output
   (modulo comments). This is non-negotiable.
2. **Human-writable.** A composer with a text editor and no viper install
   should be able to author a valid `.vip` by hand.
3. **LLM-friendly.** The format is regular enough that a language model
   can emit valid `.vip` given the grammar and two examples.
4. **Forgiving parser, strict writer.** Readers accept variable whitespace,
   optional fields, mixed case, trailing commas. Writers always emit the
   canonical form below.
5. **Order-independent where possible.** `@song` header can appear after
   `@phrase` blocks. Phrases can appear in any order; the `@song` order
   array decides playback.

## Grammar

```
file        := (line)*
line        := comment | blank | directive | data_row
comment     := '#' .* '\n'
blank       := '\n'
directive   := '@' ident (ws key_value)* '\n'
data_row    := hex2 ws cell (ws cell)* '\n'
cell        := note | '---'
note        := letter accidental octave   # e.g. C-4, D#3, F-5
accidental  := '-' | '#'
key_value   := ident '=' value
```

Notes are always 3 chars wide. `---` means empty. Sharps only (no flats,
matches every tracker from ProTracker forward). Octaves 0–9.

## Canonical example

```vip
# song.vip — generated by :gen "four-on-the-floor 140bpm"
@song  bpm=140  steps=16  order=[00,01,00,02]

@phrase 00
  # step   PU1   PU2   TRI   NOI
  00       C-4   ---   C-3   G-2
  04       D#4   ---   G-3   ---
  08       G-4   C-5   C-4   G-2
  12       F-4   ---   G-3   ---

@phrase 01
  00       ---   E-4   C-3   G-2
  02       ---   G-4   ---   ---
  04       ---   C-5   G-3   G-2
  ...

@phrase 02
  00       ---   ---   ---   G-2
  04       ---   ---   ---   G-2
  08       ---   ---   ---   G-2
  12       ---   ---   ---   G-2

@instr 00  voice=pulse25  a=02  d=08  s=0C  r=06
@instr 01  voice=triangle a=00  d=00  s=0F  r=04
@instr 02  voice=noise    a=00  d=04  s=00  r=00

# Optional modulation bindings (stage 12)
@bind sprite mario.0 scale = tri.env * 0.5 + 1.0
@bind sprite mario.0 hue   = pu1.pitch % 360
```

## Cell columns

For v0 the cell is just a note. For v0.2 (stage 3+), a cell is:

```
C-4:01:0F:A04
```

where the colon-separated fields are `note:instr:volume:effect`. Any field
may be `--` (or `---` for note) to mean empty/inherit. Whitespace between
cells is arbitrary; alignment is writer's choice.

## Canonical writer output

- Indent data rows with 2 spaces.
- Align columns to the widest cell in that column within the phrase.
- Always include a comment header row (`# step   PU1   PU2   TRI   NOI`)
  at the top of each phrase.
- Empty rows between phrases (one blank line).
- Sort directives in this order: `@song`, `@phrase` (by index), `@instr`
  (by index), `@bind`.
- Hex values lowercase in step indices, uppercase in volume/effect fields
  (tracker convention).
- One trailing newline at EOF.

## Reserved `@` directives

| directive | purpose                                    | stage |
|-----------|--------------------------------------------|-------|
| `@song`   | global song parameters                     | 3.5   |
| `@phrase` | phrase definition, followed by data rows   | 3.5   |
| `@chain`  | ordered list of phrase indices per channel | 16    |
| `@instr`  | instrument definition                      | 3     |
| `@scene`  | named snapshot of chain state              | 7     |
| `@bind`   | modulation binding                         | 12    |
| `@sprite` | sprite sheet reference + palette           | 11    |
| `@groove` | per-step timing offset table               | 16    |
| `@meta`   | free-form key=value metadata (author, etc) | any   |

## Parser error handling

- Unknown `@directive` → warn, skip the directive and its data block.
- Malformed cell → warn, treat as `---`.
- Out-of-range note → warn, clamp to 0–127.
- Duplicate `@phrase <n>` → last one wins; warn.
- Never fatal except on malformed UTF-8 or unreadable file.

The philosophy: **a partially broken `.vip` should still load and play
what it can.** Composers will hand-edit these. Being strict is hostile.

## Why not TOML/YAML/JSON?

- **TOML**: no good way to express tabular step data without ugly arrays.
- **YAML**: whitespace-sensitive in ways that conflict with tracker-style
  alignment; implementation complexity.
- **JSON**: no comments, no alignment, hostile to hand-editing.

The tracker grid is the natural representation. Formalize what composers
already write on graph paper, don't force-fit a config language.

## Validation

`viper --check song.vip` parses and reports issues without launching the
TUI. Useful for CI on a repo of `.vip` files, and for LLM output
validation before `:gen` accepts generated content.
